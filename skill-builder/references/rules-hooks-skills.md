# Rules・Hooks・Skills の違い（ホスト補足）

**位置づけ**: Agent Skills の公式仕様は **Skills** を中心に定義する。**Rules** と **Hooks** は主に **Cursor 等エディタ／エージェント製品**の機能であり、製品・バージョンで差がある。ここでは **概念の整理**と **スキル化の判断**に使う。

## 対比（一言）

| | 主な対象 | 役割のイメージ |
|---|----------|----------------|
| **Rules** | モデルへの**継続的な指示** | 「このリポジトリ／このファイル種別では常にこう振る舞え」 |
| **Hooks** | **イベント前後の自動処理** | 「このタイミングでスクリプトやチェックを実行せよ」（モデルが読む手順書ではない） |
| **Skills** | **タスク単位の手順パック**（フォルダ） | 「この種類の依頼では、この `SKILL.md` と `references/` の手順に従え」 |

## Rules（ルール）

- **典型**: Cursor の `.cursor/rules/*.mdc`（`alwaysApply` や `globs` で適用範囲を指定）。
- **向いているもの**: コーディング規約、Git／テスト方針、プロジェクト固有の前提、文体。
- **スキルとの境界**: 長大な「手順書」や **特定タスクだけ**で使う詳細ワークフローは、ルールに詰め込むより **スキル + `references/`** の方が保守しやすいことが多い（Progressive disclosure）。

## Hooks（フック）

- **典型**: Cursor の Hooks（送信前・ツール実行後などに外部コマンドを実行）。
- **向いているもの**: フォーマット、Lint、秘密情報のスキャン、ログ、ゲート、機械的な前処理／後処理。
- **注意**: **モデルへのプロンプトではない**。「フックでスキルと同じことを表現しよう」としない。役割が違う。

## Skills（スキル）

- **典型**: `SKILL.md` を核にしたディレクトリ（`references/`・`scripts/` 等）。仕様の正本は [skill-folder-spec.md](skill-folder-spec.md)。
- **向いているもの**: 繰り返し使う手順、ツール選択のナレッジ、テンプレ生成、複数ステップのワークフロー。
- **読み込みの考え方**: 多くの環境では **第1層（`name` / `description`）で候補選定**し、必要に応じて本文・`references/` を読む。`description` に手順の要約を詰めない（Description Trap）。

## スキル構築時の判断メモ

- **ずっと効かせたい短い方針** → ルール候補。
- **イベントで機械実行したい** → フック候補。
- **依頼の種類に応じて読み込ませたい手順パック** → スキル。
