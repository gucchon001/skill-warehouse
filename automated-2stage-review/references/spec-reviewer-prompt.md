# 仕様レビュアー用チェックリスト（第1段）

サブエージェント `spec-reviewer` の本文に組み込むか、起動時に Read させる。**実装の「正しさ」やリファクタの可否は扱わない**（品質レビューは第2段）。

## 入力の前提

- 対象: 変更差分、該当ファイル、**要件・仕様の参照**（Issue、仕様書、コメント、受け入れ条件）。
- 不足なら **質問を列挙**して終了し、推測で合格としない。

## 検証項目

1. **スコープ**: 変更は依頼された要件・タスクの範囲内か。余計な機能・ファイルが混ざっていないか。
2. **受け入れ条件**: 明示された条件をすべて満たすか（欠けがあれば列挙）。
3. **仕様との整合**: API 契約、データモデル、エラーハンドリング方針、UI 文言など、文書化された仕様と矛盾しないか。
4. **エッジケース**: 仕様で触れている境界条件・異常系が実装に反映されているか（仕様に無いものは「仕様外」として別扱いでよい）。
5. **セキュリティ・コンプライアンス（仕様レベル）**: 仕様で求められる認可・マスキング・保持期間等が満たされているか（詳細な脆弱性診断は別タスク）。

## 出力形式（例）

- **判定**: `PASS` | `FAIL` | `NEEDS_INFO`
- **根拠**: ファイル・行・仕様への参照つき
- **修正タスク**: FAIL のとき、優先度付きの箇条書き（品質改善ではなく仕様充足に限定）
