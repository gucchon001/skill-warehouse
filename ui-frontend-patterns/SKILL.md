# UI・フロントエンドデザインパターン

## When to Use

- ユーザーが「UI を見やすくして」「コンポーネントを揃えて」「デザインシステムを参考に」と言ったとき
- フロントエンドの見た目・操作性・一貫性を改善・レビューするとき
- ダッシュボード・一覧・フォームなど、画面の視覚的階層やコンポーネント選定を決めるとき

## 原則

### ファーストビューとトーン

- **Bold aesthetic choices**: はっきりした色・タイポグラフィ・余白の選択をする。曖昧なグレーだらけや「なんとなく」のレイアウトを避ける。
- **Distinctive but scannable**: 特徴のある見た目にしつつ、縦にスキャンできる見出し・カード・リストの階層を保つ。
- **Context-aware**: 製品の文脈（社内ツール・顧客向け・ダーク/ライト）に合った一貫したトーンにする。ジェネリックな AI 風の見た目を避ける。
- 参照: [Frontend Design Plugin](https://github.com/anthropics/claude-code/tree/main/plugins/frontend-design) の方向性。

### コンポーネントと一貫性

- **名前・役割・状態を揃える**: カード・リスト・バッジ・ナビ・ボタンなど、既存のデザインシステム（例: [Component Gallery](https://github.com/inbn/component-gallery)）のコンポーネントを参考にし、プロジェクト内で名前と役割を統一する。
- **状態を明示**: 未選択・選択中・無効・エラーなど、状態ごとにスタイルを分け、視覚的に区別できるようにする。
- **主要 CTA を accent で**: 主要なアクション（リンク・ボタン）は accent 色で一貫して強調し、二次アクションは控えめにする。

### 視覚的階層

- **見出し**: セクション見出しはフォントサイズ・ウェイトで明確にし、スキャンしやすくする。
- **余白**: ブロック単位でグルーピングされていることが分かるよう、カードやセクション間に適度な余白を入れる。
- **コントラスト**: 本文と muted テキストのコントラストを保ち、アクセシビリティ（WCAG AA 目安）を満たす。

### チェックリスト（実装・レビュー時）

- [ ] コントラスト: 文字と背景のコントラスト比が十分か
- [ ] 余白: セクション・カード間の余白でグルーピングが分かるか
- [ ] 見出し階層: h1/h2/h3 のサイズ・ウェイトが一貫しているか
- [ ] 主要 CTA: メインのリンク・ボタンが accent で強調されているか
- [ ] モバイル: 狭い幅での折り返し・タップ領域は十分か
- [ ] 状態: 無効・読み込み中・エラーなどの状態が視覚的に区別できるか

## Troubleshooting

### エラー: 手順に沿っても再現しない

**原因**: 環境・バージョン・データ依存。

**対処**: 再現手順・期待値・実際のログを 1 セットにしてから、本文の次ステップへ進む。

### エラー: テストや検証が不安定

**原因**: テスト間の共有状態、非決定的な時間・ネットワーク。

**対処**: 分離・モック・タイムアウトを見直す。
