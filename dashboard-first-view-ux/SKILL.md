# ダッシュボード・ファーストビュー UX

## When to Use

- ユーザーが「トップをわかりやすくして」「ファーストビューを改善して」「ダッシュボードの構成を見直して」と言ったとき
- 入口画面（ダッシュボード・ホーム・トップページ）のレイアウトや表示順を決めるとき
- 機能性・利便性の観点で「何を最初に見せるか」を設計するとき

## 原則

### 情報階層

- **最優先**: 「今日やるべきこと」「自分の担当」を実画面の前面に出す。Jira の「自分宛て」のように、ログイン直後に「自分がやるべき課題・タスク」が一覧で見えるようにする。
- **二次**: 説明・ヘルプ・使い方は折りたたみ、またはページ下部・トグルで開く。ファーストビューに長文説明を並べない。

### 表示順の原則

上から次の順で並べる。

1. **タイトル** ＋ 最小限のコンテキスト（1行程度）
2. **アラート**（あれば）— 遅延・エラーなど即時対応が必要なもの
3. **自分の担当** — 課題リスト・タスクリストなど、クリックで詳細へ遷移できる項目
4. **サマリメトリクス** — 過去 N 日完了／更新／作成、次の N 日期限など、数値カード
5. **ステータス内訳** — 未対応・対応中・完了などの件数
6. **遅れ・今日やるべき** — 期限切れ・本日締め切りの項目
7. **ショートカット** — 主要画面へのリンク（同期・ガイド・スケジュール・新規追加・マイページ）
8. **統計** — プロジェクト数・タスク数など（要約の直下にコンパクトにまとめてもよい）
9. **一覧・プレビュー・フィード** — プロジェクト一覧の概観、スケジュールプレビュー、最近の活動
10. **説明・使い方・機能の構成** — トグルで開閉、デフォルトは非表示

### 判断基準

- **アクション可能な項目を前に**: クリックして次の画面へ行けるもの（課題カード・リンク）を、読むだけの説明より上に置く。
- **スキャンしやすく**: 見出し・カード・リストでブロックを分け、縦に流し読みできるようにする。
- **参照**: Jira の「要約」「自分宛て」の並び、Designer Skills Collection の「構造的知識・フレームワーク」の考え方（何を必須とし、何を習慣にしないか）を参考にする。

- **UI / UX の用語定義** … グローバルスキル貯蔵庫の **[README.md の「UI / UX の定義」節](../README.md#ui-ux-definition)**（本スキルと `ui-frontend-patterns` の切り分けの正本）。

- **プロジェクト別ハブ** … 例: **`design-ui`** スキルがリポジトリごとに読み順をハブ化している場合がある。

## 実装時のチェックリスト

- [ ] タイトル直下に長文の説明を置いていないか（1行 or トグルに寄せたか）
- [ ] 「自分の担当」がファーストビュー内（スクロールなしで見える範囲）にあるか
- [ ] サマリメトリクス（完了・更新・作成・期限）が数値で一目で分かるか
- [ ] アラートはある場合のみ表示し、最優先で表示しているか
- [ ] 説明・機能の構成はトグルで開閉し、デフォルト非表示にしているか

## Troubleshooting

### エラー: ファーストビューが長く、担当タスクが画面外になる

**原因**: 説明・ヘルプ・統計を上に置き、「自分の担当」やアラートがスクロール後になっている。

**対処**: 本文「表示順の原則」に従い、**アラート → 自分の担当**を折りたたみなしで上へ。長文説明はトグルかページ下部へ移す。

### エラー: 入口が「読むだけ」でアクションが遠い

**原因**: 説明文や機能一覧が先で、クリック可能な課題・リンクが後ろ。

**対処**: 「判断基準」のとおり、**アクション可能なブロックを説明より上**に再配置する。

### エラー: プロジェクト固有の Jira レイアウトと整合を取りたい

**原因**: 抽象原則だけではチームの既存 UI と対応が曖昧。

**対処**: リポジトリに `docs/Jira参照機能の整理.md` がある場合は Read し、要約・課題リストの並びを本文の階層にマッピングする。
