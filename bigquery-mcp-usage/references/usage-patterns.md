# BigQuery MCP ユースケース別ワークフロー

ユーザーの典型的な依頼に対して、どのツールをどの順番で呼ぶかをまとめる。

---

## 1. 「どんなデータがあるか知りたい」（データカタログ探索）

ユーザーがプロジェクト内のデータ構造を把握したいとき。

### ワークフロー

```
1. list_dataset_ids          → データセット一覧を取得
2. list_table_ids(dataset)   → 興味のあるデータセットのテーブル一覧
3. get_table_info(dataset, table) → 各テーブルのスキーマを確認
4. execute_sql(SELECT * ... LIMIT 5) → サンプルデータを確認
```

### エージェントの判断ポイント

- データセットが多い場合は一覧を出してユーザーに選ばせる
- テーブルが多い場合は `search_catalog` で絞り込む
- スキーマ確認後、ユーザーに「このテーブルで何をしたいですか？」と聞く

---

## 2. 「〇〇のテーブルを探して」（テーブル検索）

テーブル名が分からないが、内容の見当はついているとき。

### ワークフロー

```
1. search_catalog(prompt=ユーザーの言葉) → テーブル候補を取得
2. get_table_info(候補のテーブル)         → スキーマで内容を確認
3. execute_sql(SELECT ... LIMIT 5)       → サンプルデータで最終確認
```

### エージェントの判断ポイント

- `search_catalog` の `prompt` にはユーザーの依頼をそのまま渡す（リライトしない）
- 候補が複数ある場合はスキーマを見比べてユーザーに提示する
- `types` フィルタを使うと絞り込みが早い（テーブルだけなら `["TABLE"]`）

---

## 3. 「このクエリを実行して」（SQL 実行）

ユーザーが SQL を提示する、または要件から SQL を組み立てるとき。

### ワークフロー

```
1. get_table_info(対象テーブル)  → スキーマを確認（カラム名・型の正確さを担保）
2. execute_sql(dry_run=true)    → スキャン量を確認（大きなテーブルの場合）
3. execute_sql(sql)             → 本実行
```

### エージェントの判断ポイント

- ユーザーが SQL を提示した場合: そのまま実行する（修正提案は別途行う）
- エージェントが SQL を組み立てる場合: 必ず先に `get_table_info` でスキーマを確認する
- 結果が大量の場合: 最初は `LIMIT 10` で実行し、ユーザーに確認してから全量を取得する
- `dry_run` で数 GB 以上のスキャンが見込まれる場合: ユーザーに確認してから実行する

---

## 4. 「このデータを分析して」（データ分析・インサイト）

ユーザーがデータに対する質問や分析を依頼するとき。

### ワークフロー

```
1. get_table_info(対象テーブル)        → スキーマを把握
2. ask_data_insights(質問, テーブル参照) → 分析・インサイト取得
3. execute_sql(補足クエリ)             → 必要に応じて追加検証
```

### エージェントの判断ポイント

- `ask_data_insights` は複雑な分析質問に強い。単純な集計は `execute_sql` で十分
- `table_references` は JSON 文字列で渡す（パラメータの型に注意）
- 複数テーブルにまたがる分析も可能（`table_references` に複数テーブルを指定）
- 結果を受けて「もっと詳しく」と言われたら、`execute_sql` で掘り下げる

### 使い分けの目安

| 依頼の種類 | 推奨ツール |
|-----------|-----------|
| 「件数を教えて」「合計は？」 | `execute_sql`（SQL で十分） |
| 「傾向を分析して」「異常値はある？」 | `ask_data_insights` |
| 「なぜこの数字が変わった？」 | `analyze_contribution` または `ask_data_insights` |

---

## 5. 「今後の数値を予測して」（時系列予測）

売上・アクセス数などの時系列データを予測するとき。

### ワークフロー

```
1. get_table_info(対象テーブル)    → タイムスタンプ列・データ列を特定
2. execute_sql(データ確認)         → 時系列データの範囲・粒度を確認
3. forecast(history_data, ...)    → 予測実行
```

### エージェントの判断ポイント

- `timestamp_col` と `data_col` の特定が最重要。スキーマから判断する
- 複数系列がある場合（店舗別・地域別など）は `id_cols` を指定する
- `horizon` はユーザーの依頼に合わせる（「来月分」なら 30、「来週分」なら 7）
- データにギャップがある場合は SQL でフィルタした履歴データを `history_data` に渡す

---

## 6. 「なぜ数値が変わったか分析して」（貢献度分析）

A/B テストの結果や、期間比較でメトリクスの変動要因を知りたいとき。

### ワークフロー

```
1. get_table_info(対象テーブル)          → スキーマを把握
2. execute_sql(テスト/対照群の確認)      → is_test_col の値・分布を確認
3. analyze_contribution(...)            → 貢献度分析を実行
```

### エージェントの判断ポイント

- `is_test_col`（テスト/対照群を区別する列）の特定が最重要
- 期間比較の場合: SQL で `CASE WHEN date >= '...' THEN true ELSE false END AS is_test` のようにフラグ列を作り、そのクエリを `input_data` に渡す
- `contribution_metric` のフォーマットに注意（`SUM(...)` / `SUM(...)/SUM(...)` の形式）
- `dimension_id_cols` は分析したい切り口（地域、デバイス、年齢層など）を指定する

---

## 7. 複合パターン: 探索 → 分析 → レポート

複数ステップにまたがる依頼のとき。

### ワークフロー例: 「売上データを探して、傾向を分析して、来月の予測を出して」

```
1. search_catalog("売上データ")          → テーブルを特定
2. get_table_info(特定したテーブル)       → スキーマ確認
3. execute_sql(基本統計)                 → データの概要を把握
4. ask_data_insights("売上の傾向")       → トレンド分析
5. forecast(...)                        → 来月の予測
6. 結果をまとめてユーザーに報告
```

### エージェントの判断ポイント

- 各ステップの結果を踏まえて次のステップを調整する
- 途中でユーザーに確認を入れる（特にテーブルの特定後、大きなクエリ実行前）
- 最終結果は見やすく整形する（表形式、箇条書き、グラフの提案など）
