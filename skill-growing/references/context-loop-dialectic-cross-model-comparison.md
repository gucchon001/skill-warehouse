# 4モデル間の弁証法比較 — 共通する思考プロセスと違い

v1（Auto）, v2（Opus 4.6）, v3（GPT-5.4）, v4（Gemini 3.1 Pro）の4つの独立思考による10回弁証法を横断比較する。

---

## 共通する思考プロセス

### 1. 「量より質／最小化」への収束

| 版 | 表現 |
|----|------|
| v1 | 最小必要量、Progressive disclosure（骨子はSKILL、詳細はreferences） |
| v2 | attentionの誘導、情報量より「注意を集中させる」 |
| v3 | 符号化効率、最小の面積で最大の指示、制約は3つに絞れ |
| v4 | 意味密度 = 方向の一貫性 ÷ トークン数、「何を書かないか」で決まる |

**共通点**: いずれも「スキルは多いほどよい」を否定し、**少なく・濃く・一貫して**という方向に収束している。

---

### 2. 検証の必須化

| 版 | 扱い方 |
|----|--------|
| v1 | 手順2「検証手順があれば実行、失敗なら修正」。検証なしはドリフトの原因。 |
| v2 | グラウンディング（実世界とのずれを検証で修正し続ける）。検証なしは「一貫して間違い続ける」。 |
| v3 | execution = ドリフト検知。検証スクリプト = ドリフト量の測定装置。 |
| v4 | 即時検証ループ（毎回）と遅延観測ループ（問題観測時）の2層。 |

**共通点**: 検証を「あればよい」ではなく**必須**とし、検証がないとスキルと実態がずれ信頼が崩れる、という構造は全版で一致している。

---

### 3. メタ（description／トリガー）の同期

| 版 | 扱い方 |
|----|--------|
| v1 | 手順4「必要ならメタを更新する」を必須。メタのドリフトは発見精度を直接落とす。 |
| v2 | 一貫性×グラウンディングの「一貫性」の一部として実態との一致。 |
| v3 | 4軸診断の Selection 診断で description/トリガー調整。 |
| v4 | 遅延観測ループで「description/トリガーが実態と合っているか確認」。 |

**共通点**: **メタを放置すると発見・一貫性が崩れる**という認識は全版にあり、更新のたびにメタを確認する必要性で一致している。

---

### 4. 重複の排除・単一情報源

| 版 | 扱い方 |
|----|--------|
| v1 | 「育て方」は skill-growing に集約。各スキルは一行だけ。重複は不整合を生む。 |
| v2 | 一貫した単一の指示。矛盾する指示の共存が最大の敵。 |
| v3 | 矛盾の排除。4軸の一貫性。 |
| v4 | チャネル間干渉（異なる推論方向の記述の混在）を密度の敵として排除。 |

**共通点**: **同じことを複数箇所に書かない／矛盾や方向のバラバラを許さない**という設計原則は全版に共通している。

---

### 5. ループの1周に「検証」を含める

| 版 | 1周の定義 |
|----|-----------|
| v1 | 読む → 検証する → 書く（追記）→ メタを更新する。 |
| v2 | 差分検知 → 読む → 検証 → 統合 → メタ更新 → 再検証。 |
| v3 | 4軸診断（Selection/Injection/Adherence/Execution）を毎回。 |
| v4 | 即時検証ループ（読む→検証→置換→再検証）＋遅延観測ループ。 |

**共通点**: **「読む→書く」の2段階では不十分**で、必ず**検証（とメタ更新）**を挟むループ設計になっている。

---

### 6. スキルの「死」・陳腐化・廃止

| 版 | 扱い方 |
|----|--------|
| v1 | 明示的な「死」の議論はなし。 |
| v2 | 古いスキルは不在より有害。「育てる」には「殺す」（アーカイブ）が含まれる。 |
| v3 | 置換で総量を減らす。「スキルは見つけた時より短くして返す」。 |
| v4 | 「育てる」には「看取る（廃止する）」が含まれる。ライフサイクルを明示管理。 |

**共通点**: v2〜v4は**廃止・アーカイブ・看取る**をループの一部として扱う。v1は手順レベルでは触れていないが、他3版は「残し続けることのリスク」を論じている。

---

### 7. 議論自体のメタ認識（仮説と検証の分離）

| 版 | 扱い方 |
|----|--------|
| v1 | なし。 |
| v2 | ラウンド10: この弁証法は実世界で検証されていない → references に置き、検証されたものだけ SKILL へ。 |
| v3 | ラウンド10: 10個の洞察を全部書くと「制約3つに絞れ」と自己矛盾 → 1つの原則だけ SKILL に昇格。 |
| v4 | ラウンド6: 自己参照の限界（外部観測だけが妥当性を判定できる）。 |

**共通点**: v2〜v4は**弁証法の結論をそのままSKILLに載せない**というメタルールに到達している（v1は手順の整理に留まりメタには触れていない）。

---

### 8. LLMの認知・機構を前提にした議論

| 版 | 前提にした機構・概念 |
|----|----------------------|
| v1 | コンテキストウィンドウのトレードオフ、retrieval でスキル全体が載る、description でスキルを選ぶ。 |
| v2 | attention の拡散、Lost in the Middle、確率的デコード、先頭・末尾重視、具体例への追従性。 |
| v3 | embedding・cos 類似度、softmax による注意の分散、post-training のデフォルト行動、パイプラインの連鎖確率。 |
| v4 | 引力場、softmax 正規化、推論のチャネリング、意味密度。 |

**共通点**: いずれも**「人間の読みやすさ＝最適」を疑い**、retrieval・attention・embedding など LLM の内部メカニズムを根拠に「少なく・一貫して・検証可能に」を導いている。

---

## モデル間の主な違い

### 1. 出発点・議論のフレーム

| 版 | 第1ラウンドの問い | 性格 |
|----|-------------------|------|
| v1 | スキル文脈の「量」 | 手順・運用の観点から列挙的に議論（量→トリガー→検証→重複→メタ→主体→分離→失敗時→依存→1周）。 |
| v2 | スキルとは何か | 概念の定義から始め、attention・差分・矛盾・グラウンディング・死・主体・ネットワーク・暗黙知・議論の限界へ。 |
| v3 | スキルの「性能」とは何を測っているか | 指標の定義から始め、selection→injection→adherence→execution のパイプラインを導入。工学的・システム論的。 |
| v4 | スキルは「記憶」か「プログラム」か | 存在論的な問いから始め、チャネリング→引力場→意味密度→干渉→自己参照→遅延→観測→ライフサイクル→二重の読者へ。 |

**違い**: v1は**手順の網羅**、v2は**概念の掘り下げ**、v3は**測定可能なパイプライン**、v4は**存在論・認識論**に近いレイヤーで議論している。

---

### 2. 中核メタファー・モデル

| 版 | 中核メタファー・モデル |
|----|------------------------|
| v1 | なし（「手順」と「トレードオフ」の素直な議論）。 |
| v2 | attention の誘導、差分駆動、一貫性×グラウンディング。 |
| v3 | **パイプライン**（selection × injection × adherence × execution の連鎖確率）、adherence がサイレントなボトルネック、「従われる確率」。 |
| v4 | **引力場**、**意味密度**（方向の一貫性 ÷ トークン数）、チャネル間干渉、二重の読者。 |

**違い**: v1はメタファーをほとんど使わず、v2は「attention・差分・グラウンディング」、v3は「パイプラインと確率」、v4は「場・密度・チャネル」という**物理・工学的な比喩**を前面に出す。

---

### 3. 「最大の敵」の置き方

| 版 | 最も性能を落とすもの（第1位に近いもの） |
|----|------------------------------------------|
| v1 | コンテキストの肥大化、検証の欠如、重複とメタのドリフト（3つ並列）。 |
| v2 | **矛盾する指示の共存**（質の問題；不足より矛盾が致命的）。 |
| v3 | **Adherence のサイレントな劣化**（書く量が増えるほど追従確率が下がる）。 |
| v4 | **チャネル間干渉**（矛盾だけでなく「推論方向の違う記述」の混在）。 |

**違い**: v1は**量・検証・重複**を並列、v2は**矛盾**、v3は**adherence の見えにくい劣化**、v4は**方向の混在（チャネル間干渉）**を最悪の敵として強調している。

---

### 4. ループの主体・トリガー

| 版 | 誰が／何がループを回すか |
|----|---------------------------|
| v1 | エージェントが「スキルに手を入れるたびにこのスキルを読む」ことで、ループの主体はエージェント。 |
| v2 | 理想は**自動トリガー**（検証失敗の自動検知）。現状は人間の「育てて」に依存する構造的弱点を指摘。 |
| v3 | 4軸診断を毎回行う。追記ではなく**置換**が基本。 |
| v4 | **即時検証ループ**（execution）と**遅延観測ループ**（selection/adherence）の2層。観測基盤が欠けている現実を明示。 |

**違い**: v1は「エージェントが読む」で十分とし、v2は**自動トリガーの欠如**を問題にし、v4は**観測基盤の欠如**まで言及している。v3は主体より**操作（置換）**を強調している。

---

### 5. 追記 vs 置換

| 版 | 書くときの基本操作 |
|----|--------------------|
| v1 | 知見を**追記**する（SKILL または references）。 |
| v2 | 矛盾の排除＋既知の削除＋知見の追記。置換の明示は弱い。 |
| v3 | **「追記」ではなく「置換」**を基本操作。総量は一定か減少。 |
| v4 | 即時検証ループで「**置換（追記ではなく）**」。 |

**違い**: v1は追記を前提にしている。v2は「矛盾・既知の削除」で置換に近いが、v3・v4は**置換を明示的な原則**として打ち出している。

---

### 6. 数値・具体性

| 版 | 数値・具体例 |
|----|--------------|
| v1 | 手順4つ、ラウンド10個。数値の指定はなし。 |
| v2 | 現状 vs 理想の表。数値はなし。 |
| v3 | **「制約は3つに絞れ」**、4段階パイプライン、4軸診断の表。 |
| v4 | **意味密度 = 方向の一貫性 ÷ トークン数**、密度の敵3つ。 |

**違い**: v3は**「3つ」**という具体的な数と診断表を出し、v4は**式**で密度を定義している。v1・v2は数値や式を出さず、定性論に留まっている。

---

### 7. スキルの境界の定義

| 版 | スキルの境界 |
|----|--------------|
| v1 | 重複を避け skill-growing に集約。境界の明示的な定義はなし。 |
| v2 | 疎結合だが入出力契約で連携。ネットワーク効果。 |
| v3 | パイプラインの4段階（selection/injection/adherence/execution）。境界は段階として。 |
| v4 | **スキルの境界 = 推論の方向が変わる地点**。問題空間が似ていても推論方向が違えば別スキル。 |

**違い**: v4だけが**「推論の方向が変わる地点」**という明示的な境界定義を出している。他は「重複しない」「段階を分ける」「入出力でつなぐ」程度。

---

### 8. 読者・ユーザーの明示

| 版 | スキルの読者 |
|----|--------------|
| v1 | エージェント（「このスキルを読む」）。人間への言及は薄い。 |
| v2 | エージェント。人間は「育てて」と言うトリガーとして。 |
| v3 | エージェント（4段階のパイプライン）。人間は暗黙的。 |
| v4 | **二重の読者**：エージェント（実行時）と人間（検証・育成時）。実行指示はエージェント最適化、説明・根拠は人間最適化で分離。 |

**違い**: **「誰のための文書か」を明示的に論じているのは v4 だけ**で、実行指示と説明の分離まで言及している。

---

## まとめ

- **共通していること**: 量の最小化・検証の必須化・メタの同期・重複・矛盾（および方向のバラバラ）の排除・ループに検証を含める・スキルの死／廃止の扱い（v2〜v4）・議論の結論をそのままSKILLに載せないメタ認識（v2〜v4）・LLMの機構を根拠にした設計、の各点で4版は一致している。
- **違うこと**: 出発点（手順 vs 概念 vs 指標 vs 存在論）、中核メタファー（なし vs attention/差分 vs パイプライン vs 引力場・密度）、最大の敵（量/検証/重複 vs 矛盾 vs adherence の劣化 vs チャネル間干渉）、ループの主体・トリガー・観測の現実、追記か置換か、数値・式の有無、スキル境界の定義の有無、読者（二重の読者）の明示の有無、で差がある。

この比較を references に置いておくことで、どの版のどの論点を SKILL や運用に昇格させるかの判断材料にできる。
