# 議論の活かし方 — スキル育成を自律成長させる提案

**前提**: 各モデル（v1 Auto, v2 Opus, v3 GPT, v4 Gemini）に対して「コンテキストループをどう回すか、性能を落とす本質と引っ張る本質は何か」を 10 回弁証法で問うた結果が `context-loop-dialectic*.md` と `context-loop-dialectic-cross-model-comparison.md` にある。本ドキュメントは、その議論をどう運用に落とせば **skill-growing 自体が使うほど育ち**、さらに **gws のような不安定スキルを安定化できるか**を整理した提案である。

---

## 1. 結論

**skill-growing は「知見を貯めるスキル」ではなく、「失敗を観測して、矛盾を除去し、実証済みの正規ルートだけを残す制御系スキル」として育てるのがよい。**

特に Google 系の不安定さに対しては、次の4点を中心に据える。

1. **追記型ではなく置換型で育てる**  
   失敗例や回避策を足し続けるのではなく、正規ルートを差し替える。
2. **4軸で診断する**  
   `selection / injection / adherence / execution` のどこで壊れたかを切り分ける。
3. **対象 × 操作 × チャネルで正規ルートを一意にする**  
   例: Sheets 読取、Sheets 書込、Docs 読取、Docs 書込を同じスキルで曖昧に扱わない。
4. **仮説と証拠を分ける**  
   弁証法の洞察は references に置き、複数回の成功で初めて SKILL に昇格させる。

---

## 2. 各モデルの本質的意図

### 2.1 v1（Auto）

- **焦点**: 手順の欠落を防ぐこと。
- **得られる原則**: 「読む → 検証する → 書く → メタ更新」の 1 周を欠かさない。
- **弱み**: 追記前提であり、手順自体が古くなったときにどう置き換えるかが弱い。

### 2.2 v2（Opus）

- **焦点**: 一貫性とグラウンディング。
- **得られる原則**: 最大の敵は知識不足ではなく **矛盾する指示の共存**。弁証法の結論は仮説として references に置き、検証されたものだけを SKILL に昇格させる。
- **弱み**: 自動トリガーや観測基盤の具体設計までは踏み込まない。

### 2.3 v3（GPT）

- **焦点**: 性能を歩留まりで見ること。
- **得られる原則**: スキル性能は `selection × injection × adherence × execution` の連鎖確率。最大のサイレントボトルネックは **adherence 劣化**。追記ではなく **置換**、制約は 3 つ以内。
- **弱み**: 何を昇格させるかの選定には実証が必要。

### 2.4 v4（Gemini）

- **焦点**: スキルは推論のチャネリングであること。
- **得られる原則**: 最大の敵は **チャネル間干渉**。推論方向が違う内容を1つに混ぜると意味密度が下がる。即時検証と遅延観測を分ける。
- **弱み**: 観測基盤がないと理想形まで届かない。

### 2.5 比較から見える統合解

- v1 は **ループを欠かすな** と言う。
- v2 は **矛盾を残すな、仮説をそのまま昇格させるな** と言う。
- v3 は **従われる確率を上げろ、追記ではなく置換せよ** と言う。
- v4 は **推論方向を混ぜるな、観測できないものは改善できない** と言う。

この4つを統合すると、skill-growing の役割は「情報を足すこと」ではなく、**不安定化の原因を観測し、正規ルートを収束させること**になる。

---

## 3. なぜこの提案になるのか

### 3.1 最大の問題は「知識不足」ではない

比較ドキュメントでは、性能低下の主因がモデルごとに次のように表れている。

- v2: **矛盾する指示の共存**
- v3: **adherence のサイレント劣化**
- v4: **チャネル間干渉**
- v1: **検証の欠如とメタのドリフト**

これらはすべて、「もっと説明を書く」ことではなく、**何を正規ルートとして残すかが曖昧**であることに収束する。  
したがって、skill-growing に必要なのは知見の追加よりも **ルートの整理と淘汰** である。

### 3.2 gws の不安定さはこの構造に一致する

たとえば Google 系の操作では、現実に次のような症状が起こりうる。

- スプレッドシートは読めるが書込で壊れる
- Sheets は動くが Docs は動かない
- Windows では壊れるが別環境では動く

これは単なる execution 失敗だけでなく、

- 同じスキルに複数の操作が混在している
- どのチャネルを正規ルートとするかが不統一
- 手順を追記し続けて本体が肥大化している

という、v2/v3/v4 の指摘が同時に起きている状態と見なせる。

### 3.3 自律成長には「自己参照」ではなく「外部観測」が必要

v4 が示す通り、skill-growing 自体の妥当性は、自分を読むだけでは検証できない。  
正しさは **他スキルを育てた結果** でしか測れない。  
つまり、`gws` が安定したか、`sheet` 系スキルが迷わず選ばれるようになったか、といった **外部結果** が skill-growing の学習データになる。

## 4. 提案A: skill-growing を「使うほど育つ」形に変える

### 4.1 仮説プールと証拠ログを分ける

- **仮説プール**: 既存の弁証法ドキュメント群。モデルごとの洞察や未検証の原則を置く場所。
- **証拠ログ**: 実際にどのスキルにどの原則を適用し、何が成功し何が失敗したかを書く場所。

証拠ログには最低限次を書く。

- 対象スキル
- 発生した問題
- 適用した原則
- 変更前の正規ルート
- 変更後の正規ルート
- 検証方法
- 結果
- SKILL に昇格したか、references に留めたか

### 4.2 昇格ルールを明示する

- **references に残すもの**: 1回しか試していない仮説、失敗理由の分析、モデル間で競合する見解。
- **SKILL に昇格するもの**: 複数回の適用で一貫して成功した、短く強い原則。

昇格時の基準は次の通り。

1. 同じ原則で複数回成功している
2. 他の原則と矛盾しない
3. 1文か短い箇条書きで言える
4. 実行時の判断を明確に減らせる

### 4.3 基本操作を「追記」から「置換」に変える

運用ルールとして、次を原則にする。

- 失敗した正規ルートは追記ではなく**置換**する
- 古い説明が新しい説明と競合するなら古い方を削る
- references は厚くてもよいが、SKILL 本体は短く保つ

この方針は v3 の adherence 最適化と、v4 の意味密度最適化の両方に合う。

### 4.4 ループのトリガーを「人間の依頼」から「検証結果」に寄せる

理想は v2 の言う自動トリガーだが、現実的には次の段階的運用がよい。

1. 各スキルに検証手順を明記する
2. 検証失敗時は skill-growing を読むと明示する
3. 更新後に再検証する
4. 成否を証拠ログに残す

これで、人間の「育てて」だけでなく、**失敗という差分** がループの起点になる。

## 5. 提案B: Google 系スキルを安定させる

### 5.1 「対象 × 操作 × チャネル」で正規ルートを決める

Google 系の不安定さは、「Google を扱う」という広い問題空間を1つにまとめすぎると起きやすい。  
v4 の原則に従えば、**推論方向が変わる地点** で境界を切るべきである。

推奨する整理単位は次の通り。

- `sheets.read`
- `sheets.write`
- `docs.read`
- `docs.write`
- `auth / scope`
- `cli params encoding`

そして各能力に対して **canonical channel** を1つ決める。

例:

| capability | canonical channel | fallback | 備考 |
|------------|-------------------|----------|------|
| `sheets.read.values` | `gws` または API | あり | 読取は複数許容でもよい |
| `sheets.write.values` | Google Sheets API | なし or 限定 | 書込は1つに固定した方が安定しやすい |
| `docs.read` | `gws` | あり | Docs 系は別方向として分離 |
| `cli-json-params` | `gws-params-encoding` | なし | これは補助スキルであり、書込の正規チャネルではない |

### 5.2 4軸診断を適用する

| 軸 | 問い | Google 系での見方 |
|----|------|------------------|
| Selection | 正しい依頼で正しいスキルが選ばれるか | Sheets 書込で `gws` 系に流れていないか |
| Injection | 情報量が他文脈を圧迫していないか | 1スキルに Sheets/Docs/auth/params が詰まりすぎていないか |
| Adherence | 書いてある強い制約に本当に従っているか | 「書込は API のみ」のような制約が別スキルで破られていないか |
| Execution | 現在の環境で動くか | OS、shell、scope、認証、対象 ID を含めて再現できるか |

### 5.3 即時検証と遅延観測を分ける

- **即時検証**: 実行時にすぐ確認できるもの
  - 認証できるか
  - スコープが足りるか
  - 対象が存在するか
  - 最小読取・最小書込が成功するか
- **遅延観測**: 後からしか分からないもの
  - そのスキルが適切に選ばれていたか
  - 手順が長すぎて守られなかったか
  - ユーザーが同じ失敗に何度も遭遇していないか

### 5.4 失敗時は「回避策の追記」ではなく「正規ルートの置換」

失敗したときの基本動作は次の通り。

1. どの capability が失敗したかを特定する
2. 4軸のどこで壊れたかを分類する
3. 失敗した正規ルートを見直す
4. 必要なら別チャネルを canonical に昇格させる
5. 古い正規ルートの記述は削るか relegated note に落とす

これで v3 の「追記型ループの振動」を避けられる。

## 6. gws を題材にした具体運用

### 6.1 まずやること

1. Google 系スキルを棚卸しする
2. 各スキルを `対象 × 操作 × チャネル` で分類する
3. 正規ルート表を1枚作る
4. 各 capability に最小検証を用意する
5. 失敗ログの書式を統一する

### 6.2 失敗ログの最小項目

- 問題が起きた依頼
- 選ばれたスキル
- 対象サービス
- 操作種別
- 実行チャネル
- OS / shell
- 認証スコープ
- 実際のエラー
- 再現手順
- 成功した代替経路
- 正規ルートを置換したか

### 6.3 skill-growing の1周として回す手順

1. 対象スキルと references を読む
2. 検証を実行する
3. 失敗を 4軸に分類する
4. 正規ルートを置換する
5. description と trigger を実態に合わせる
6. 再検証する
7. 証拠ログに残す

これにより、gws の安定化そのものが skill-growing の学習材料になる。

---

## 7. 次に弁証法で問うとよいこと

議論を続けるなら、次は次の問いが有効である。

- 証拠ログが増えたとき、SKILL に昇格させる原則をどう選ぶか
- canonical channel を変える閾値をどう決めるか
- 複数モデルの結論が競合したとき、実証でどちらを採るか
- 「追記ではなく置換」をやりすぎて有用な履歴を失わないための境界は何か

これらの結論も、まずは references に置き、実証で効いたものだけを SKILL に昇格させる。

---

## 8. まとめ

| 目的 | 提案 | 根拠 |
|------|------|------|
| skill-growing を自律成長させる | 仮説プールと証拠ログを分け、実証済み原則だけ昇格させる | v2, v4 |
| スキルの密度を上げる | 追記ではなく置換を基本にする | v3, v4 |
| 不安定スキルを安定化する | 4軸診断で壊れ方を分類する | v3 |
| Google 系の混線を防ぐ | `対象 × 操作 × チャネル` で canonical route を定義する | v4, v2 |
| 人手依存を減らす | 検証失敗をループの起点にする | v1, v2 |

この提案自体もまだ仮説であり、references に置いて実運用で検証し、効いた原則だけを skill-growing の SKILL 本体へ昇格させる。
