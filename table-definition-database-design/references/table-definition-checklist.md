# テーブル定義書 チェックリスト

テーブル定義書を作成・更新するときに確認する項目。スキル「table-definition-database-design」から参照する。

---

## 1. ドキュメント構成

- [ ] 対象スキーマ・対象機能の範囲が明記されている
- [ ] 既存ドキュメントとの関係（新規セクション／独立ドキュメント）が書いてある
- [ ] 更新日・目的が書いてある（任意だが推奨）

---

## 2. エンティティ・テーブル一覧

- [ ] 論理テーブル名・説明・役割が一覧になっている
- [ ] 主キー（PK）が各テーブルで明示されている
- [ ] 他テーブルとの関係（1 対 1 / 1 対多 / 多対多）が分かる
- [ ] 複数テーブルに分割する場合、分割理由（個人情報の分離・権限・拡張性等）が簡潔に書いてある

---

## 3. 物理定義（各テーブル）

- [ ] 全カラムに**データ型**が書いてある（TEXT / VARCHAR(n) / INTEGER / DATE / TIMESTAMPTZ / UUID / JSONB 等）
- [ ] NULL 許容か NOT NULL かが分かる
- [ ] 制約: PRIMARY KEY, UNIQUE, CHECK, DEFAULT, REFERENCES が書いてある
- [ ] 複数選択やコード列は保存形式（カンマ区切り TEXT / 配列型 / 中間テーブル）が書いてある
- [ ] インデックス（検索・結合で使う列）が列挙されている
- [ ] COMMENT または説明で、業務上の意味が分かる

---

## 4. 氏名・複合項目の方針

- [ ] 氏名を「姓・名・姓カナ・名カナ」で持つか、「氏名・氏名カナ」の 2 項目で持つかが書いてある
- [ ] フォーム UI は 2 入力（氏名・氏名カナ）で送信時に分割するか、4 入力（姓・名・姓カナ・名カナ）にするかが書いてある
- [ ] 他システムへの連携方針が書いてある（そのまま渡す／結合して渡す／連携時に分割する）

---

## 5. 設定値一覧

- [ ] 選択肢を持つ項目ごとに「コード・表示ラベル・説明」の表がある
- [ ] コードの許容値が DB では CHECK または参照で限定されることが定義書に書いてある
- [ ] フォーム・API・DB で同じコード体系を使うことが明記されている

---

## 6. RLS・セキュリティ（必要な場合）

- [ ] RLS を有効にするテーブルが列挙されている
- [ ] ポリシー（誰が SELECT / INSERT / UPDATE / DELETE できるか）の概要が書いてある
- [ ] サービスロールで挿入する API 等は、その旨が定義書または設計メモに書いてある

---

## 7. マイグレーション・型同期

- [ ] 新規テーブルはマイグレーションファイルに SQL で追加する方針が守られる
- [ ] 既存テーブル破棄時は、DROP と新 CREATE の順序・データ移行の要否が書いてある
- [ ] マイグレーション適用後に型生成を実行することが記載されている
