# 開発計画・仕様とコードの突合せ観点（チェックリスト）

エージェントはプロジェクトに応じて項目を取捨選択する。すべてを毎回やる必要はない。

## 1. ドキュメント同士

- [ ] 開発計画に書いたマイルストーンが、別ファイルの仕様・ロードマップと日付・名称で一致している
- [ ] 「完了」「対応済み」が、実際のリリースノートや Issue の状態と矛盾しない
- [ ] 同じ用語（サービス名・モジュール名）が doc 間で揃っている

## 2. 公開 URL・ルート・API

- [ ] 計画・仕様に記載のパス・HTTP メソッドが、ルータ定義（例: Flask `@app.route`、FastAPI、Next.js `app/`）と一致
- [ ] 廃止したエンドポイントが doc に残っていない

## 3. 設定・環境

- [ ] 計画・README に載る環境変数名が、コード（`os.environ` / `process.env` / 設定モジュール）に存在する
- [ ] デフォルト値や単位（MB 秒など）が doc とコードで一致

## 4. 依存・ランタイム

- [ ] 言語・ランタイム・主要ライブラリのバージョン帯が、`requirements.txt` / `pyproject.toml` / `package.json` と矛盾しない
- [ ] 「使用しない」と計画に書いた技術が、まだ依存に残っていないか（逆も）

## 5. データ・外部サービス

- [ ] 計画に書いた DB・ストレージ・外部 API が、接続コード・マイグレーションの存在と矛盾しない
- [ ] スキーマ名・テーブル名を doc が断言している場合、マイグレーションと一致

## 6. 非機能・制限

- [ ] アップロード上限・タイムアウト・レート制限などが、コードの定数と doc で一致
- [ ] セキュリティ上の前提（認証方式・公開範囲）が実装と食い違わない

## 7. テスト・CI（任意）

- [ ] 計画に「必須テスト」や CI ゲートの記述がある場合、実際の `pytest` / `npm test` / GitHub Actions 等と名前・パスが一致している

## 出力時の書き方

各項目について **一致 / 不一致 / 未確認** のいずれかと、根拠（ファイルパスまたは記述箇所）を短く付ける。
