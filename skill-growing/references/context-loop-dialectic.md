# スキルを育てるスキルとコンテキストループ — 10回弁証法（v1）

**モデル**: Auto（初回）。v2=Opus 4.6, v3=GPT-5.4, v4=Gemini 3.1 Pro はそれぞれ独立思考で別ファイルにあり、本ファイルは参照していない。

**目的**: エージェントスキルの「性能を落とす本質」と「性能を引っ張る本質」を、LLM開発者視点で弁証法とクリティカルな問いで抽出し、**スキルを育てるスキル自体**のコンテキストループの回し方を結論する。

---

## 前提定義（LLM開発者視点）

### スキル「性能」の定義

| 軸 | 内容 |
|----|------|
| **発見** | 適切なトリガーでスキルが選択されること |
| ** retrieval 効率** | 必要な文脈だけがコンテキストに載り、ノイズが少ないこと |
| **実行品質** | 手順に従ったときに再現可能で意図通り動くこと |
| **保守コスト** | 更新・検証が一貫した手順で行え、ドリフトしにくいこと |

### コンテキストループの定義

「スキルを育てる」ために、**読む → 検証する → 書く → メタを更新する** のサイクルを回すこと。その際に「何をいつ読むか」「何を検証するか」を決めるのがコンテキストループの設計。

---

## 本質の仮説（議論前）

- **性能を落とす本質**: 文脈の肥大化、曖昧なトリガー、検証の欠如、重複・矛盾した記述。
- **性能を引っ張る本質**: 最小必要コンテキスト、明確なトリガー、検証可能な手順、単一の信頼できる情報源。

以下、これらを弁証法で検証・深化する。

---

## 第1ラウンド：スキル文脈の「量」

| 項目 | 内容 |
|------|------|
| **テーゼ** | スキルは「多いほど親切」で性能が上がる。手順を細かく書けばエージェントの失敗が減る。 |
| **クリティカルな問い** | コンテキストウィンドウに載る「他」のトークンとのトレードオフで、本当に総合性能は上がるか？ |
| **アンチテーゼ** | スキルが長いと retrieval でスキル全体が載り、会話の文脈が圧迫される。重要でない記述が決定を鈍らせる。 |
| **総合** | **最小必要量**が性能を引っ張る。Progressive disclosure（SKILL.md は骨子、details は references）が本質。 |

---

## 第2ラウンド：トリガー・発見

| 項目 | 内容 |
|------|------|
| **テーゼ** | description とトリガーを多く書けば、スキルが選ばれやすくなる。 |
| **クリティカルな問い** | トリガーが広すぎると、無関係な会話でもスキルが発動し、ノイズになるのでは？ |
| **アンチテーゼ** | 曖昧・広いトリガーは誤発動を増やし、ユーザー意図とずれたスキル適用で性能が落ちる。 |
| **総合** | **明確で狭いトリガー**（「スキルを更新して」「検証を追加して」など）が発見精度を上げ、性能を引っ張る。 |

---

## 第3ラウンド：検証の有無

| 項目 | 内容 |
|------|------|
| **テーゼ** | 検証は「あればよい」で、手順だけ書いておけば十分。 |
| **クリティカルな問い** | 検証がないと、スキル更新後に誰が「正しさ」を保証するか？ |
| **アンチテーゼ** | 検証がないスキルは、時間経過で記述と実態がずれ（ドリフト）、実行品質が確実に落ちる。 |
| **総合** | **検証可能な手順**（コマンド・スクリプト・チェックリスト）をスキルに紐づけることが、性能維持の本質。 |

---

## 第4ラウンド：重複と単一情報源

| 項目 | 内容 |
|------|------|
| **テーゼ** | 各スキルに「育て方」を書いておけば、そのスキル単体で完結して便利。 |
| **クリティカルな問い** | 同じ内容を複数スキルに書くと、更新時に一貫性をどう保つか？ |
| **アンチテーゼ** | 重複は必ず不整合を生む。一箇所だけ直して他を忘れると、性能（再現性・信頼）が落ちる。 |
| **総合** | **「育て方」は skill-growing に集約**し、各スキルは「このスキルの更新は skill-growing に従う」と一行だけ書く。単一の信頼できる情報源が性能を引っ張る。 |

---

## 第5ラウンド：メタデータの更新

| 項目 | 内容 |
|------|------|
| **テーゼ** | 本文を更新すればよく、name/description はあまり変えなくてよい。 |
| **クリティカルな問い** | description が実態とずれていると、発見（retrieval）でどう影響するか？ |
| **アンチテーゼ** | メタのドリフトは発見精度を直接落とす。LLM は description でスキルを選ぶため、古い description は誤選択を招く。 |
| **総合** | **手順4「必要ならメタを更新する」は必須**。スキルを更新したら「トリガー・description が実態と合っているか」を毎回確認する。 |

---

## 第6ラウンド：コンテキストループの「誰が回すか」

| 項目 | 内容 |
|------|------|
| **テーゼ** | コンテキストループは人間が手動で回すものでよい。 |
| **クリティカルな問い** | エージェントがスキルを更新するたびに、このスキルを「読む」だけでループは回るか？ |
| **アンチテーゼ** | 人間だけに頼ると更新頻度が落ち、スキルが古くなる。エージェントが「スキルに手を入れるたびにこのスキルを読む」と明示することで、ループの主体がエージェントになる。 |
| **総合** | **ループを回す主体はエージェント**。SKILL に「スキルに手を入れるたびにこのスキルを読む」と書くことで、毎回コンテキストに skill-growing が載り、手順が適用される。 |

---

## 第7ラウンド：references の分離

| 項目 | 内容 |
|------|------|
| **テーゼ** | すべて SKILL.md に書いた方が、1ファイルで完結して読みやすい。 |
| **クリティカルな問い** | 1ファイルに詰めると、retrieval 時に「全部載せる」か「載せない」かの二値になり、トークン効率が悪化するのでは？ |
| **アンチテーゼ** | 長い SKILL は「スキル選択時」に全文がコンテキストに入り、会話の余地を奪う。本質は「骨子は SKILL、詳細は references」の分離。 |
| **総合** | **Progressive disclosure**。SKILL は手順の要約と参照リンクに留め、長い説明・弁証法・履歴は references に置く。これが retrieval 効率と実行品質の両方を引っ張る。 |

---

## 第8ラウンド：検証の「失敗時」の扱い

| 項目 | 内容 |
|------|------|
| **テーゼ** | 検証が失敗したら「とりあえずスキルを更新して、あとで直す」でよい。 |
| **クリティカルな問い** | 失敗を放置すると、スキルと実世界の不整合が残り、次回から実行品質が落ちるのでは？ |
| **アンチテーゼ** | 検証失敗を許容すると、スキルは「書いてあるが動かない」状態になり、信頼が崩れる。 |
| **総合** | **検証失敗時は修正を完了してから追記する**。手順2「失敗があれば修正する」を必須にし、失敗をスキルに残すなら「回避策・失敗例」として明示する。 |

---

## 第9ラウンド：スキル同士の依存

| 項目 | 内容 |
|------|------|
| **テーゼ** | スキルは互いに独立である方が、依存がなくてシンプル。 |
| **クリティカルな問い** | 更新手順を「skill-growing」に集約すると、skill-growing が壊れたら全スキルの育て方が分からなくなるのでは？ |
| **アンチテーゼ** | 依存を避けようとして各スキルに重複を書くと、不整合で性能が落ちる。一方、skill-growing は「メタスキル」として薄く保ち、参照だけにすれば単一障害点のリスクは小さい。 |
| **総合** | **メタスキルへの依存は許容し、メタスキルは薄く保つ**。skill-growing は「手順＋references へのリンク」だけにし、深い議論は references（本ドキュメントなど）に隔離する。 |

---

## 第10ラウンド：コンテキストループの「1周の定義」

| 項目 | 内容 |
|------|------|
| **テーゼ** | ループは「読む → 書く」の2段階で十分。 |
| **クリティカルな問い** | 検証なしで「書く」だけだと、書いた内容の正しさをどう保証するか？ |
| **アンチテーゼ** | 検証を飛ばすと、書いた手順が実行できないままスキルに残り、性能を落とす。 |
| **総合** | **1周 = 読む → 検証する → 書く（追記）→ メタを更新する**。この4ステップを欠かさないことが、コンテキストループで性能を落とさない本質。 |

---

## プロセス可視化

### 弁証法ループの流れ（1ラウンド）

```
[ テーゼ ]  →  [ クリティカルな問い ]  →  [ アンチテーゼ ]  →  [ 総合 ]
     ↑                                                                  |
     |__________________________ 次のラウンドのテーゼや前提に ___________|
```

### 10ラウンドの対応関係

| ラウンド | 論点 | 性能を落とす要因（総合の対極） | 性能を引っ張る要因（総合） |
|----------|------|--------------------------------|-----------------------------|
| 1 | 文脈の量 | 過剰な記述・長文 | 最小必要量・Progressive disclosure |
| 2 | トリガー | 曖昧・広いトリガー | 明確で狭いトリガー |
| 3 | 検証 | 検証なし | 検証可能な手順の紐づけ |
| 4 | 重複 | 各スキルに重複記述 | 単一情報源（skill-growing に集約） |
| 5 | メタ | メタの放置 | 更新時のメタ確認・更新 |
| 6 | ループの主体 | 人間任せ | エージェントが毎回このスキルを読む |
| 7 | 分離 | 全部 SKILL に詰める | SKILL は骨子、details は references |
| 8 | 失敗時 | 検証失敗を放置 | 失敗時は修正してから追記 |
| 9 | 依存 | 重複で依存回避 | メタスキルに薄く依存 |
| 10 | 1周の定義 | 読む→書くだけ | 読む→検証→書く→メタ更新 |

### コンテキストループの回し方（結論）

```
                    ┌─────────────────────────────────────┐
                    │  エージェントが「スキルを更新する」  │
                    │  というタスクを検知                  │
                    └─────────────────┬───────────────────┘
                                      │
                                      ▼
                    ┌─────────────────────────────────────┐
                    │  1. skill-growing を読む（必ず）     │  ← コンテキストに載せる
                    │     → 該当スキル + references を読む │
                    └─────────────────┬───────────────────┘
                                      │
                                      ▼
                    ┌─────────────────────────────────────┐
                    │  2. 検証手順があれば実行             │
                    │     → 失敗なら修正してから先へ       │
                    └─────────────────┬───────────────────┘
                                      │
                                      ▼
                    ┌─────────────────────────────────────┐
                    │  3. 知見を追記（SKILL または         │
                    │     references、長いのは references）│
                    └─────────────────┬───────────────────┘
                                      │
                                      ▼
                    ┌─────────────────────────────────────┐
                    │  4. メタ（description/トリガー）     │
                    │     が実態と合うか確認し、必要なら更新│
                    └─────────────────┬───────────────────┘
                                      │
                                      └──────→ 次の「スキル更新」タスクで 1 に戻る
```

**コンテキストループを回す具体的な仕掛け**:

- **エージェント側**: 「スキルに手を入れるたびにこのスキルを読む」という指示が SKILL にあり、かつ「スキルを更新して」「スキルを育てて」「検証を追加して」がトリガーになる。
- **毎回 1 が実行される**: タスク発動時に skill-growing が retrieval で選ばれ、その結果「該当スキルを読み直す」「references を読む」がコンテキストに載る。
- **2→3→4 を欠かさない**: スキル本文に手順として明示されているため、エージェントは検証→追記→メタ確認のループを踏む。

## 最終結論

### エージェントスキルの性能を最も落とす本質（3つ）

1. **コンテキストの肥大化** — 必要以上の記述を載せ、retrieval と実行の両方でノイズになる。
2. **検証の欠如** — 検証がないとスキルと実態がずれ、実行品質と信頼が確実に落ちる。
3. **重複とメタのドリフト** — 複数箇所に同じことを書くこと、および description/トリガーを更新しないことで、発見と一貫性が崩れる。

### エージェントスキルの性能を最も引っ張る本質（3つ）

1. **最小必要コンテキストと Progressive disclosure** — SKILL は骨子と参照だけ、詳細は references に分離する。
2. **検証可能な手順の紐づけ** — 各スキルに「検証は ○○ を実行」と書いてあり、失敗時は修正してから追記する。
3. **単一の信頼できる情報源とメタの同期** — 「育て方」は skill-growing に集約し、更新のたびにメタを実態に合わせる。

### スキルを育てるスキルにおけるコンテキストループの回し方

- **ループの1周**: 読む（skill-growing ＋ 該当スキル ＋ references）→ 検証する → 書く（追記）→ メタを更新する。
- **誰が回すか**: エージェント。SKILL に「スキルに手を入れるたびにこのスキルを読む」と書くことで、更新タスクのたびにこの手順がコンテキストに載る。
- **性能を落とさないために**: 手順を省略しない（特に検証とメタ更新）。長い議論・履歴は references に置き、SKILL は薄く保つ。

---

## 議論プロセス要約

1. **量 vs 最小化** → 最小必要量が性能を引っ張る。
2. **トリガー広く vs 狭く** → 明確で狭いトリガーが発見を安定させる。
3. **検証任意 vs 必須** → 検証可能な手順の紐づけが品質を維持する。
4. **重複許容 vs 単一情報源** → 育て方は skill-growing に集約する。
5. **メタ放置 vs 更新** → 更新時にメタを合わせる。
6. **人間が回す vs エージェントが回す** → エージェントが毎回読む設計にする。
7. **1ファイル vs 分離** → Progressive disclosure で references に詳細を分離する。
8. **失敗放置 vs 修正してから追記** → 検証失敗時は必ず修正する。
9. **依存ゼロ vs メタスキルに薄く依存** → メタスキルは薄く保ち依存を許容する。
10. **2段階 vs 4段階** → 1周は「読む→検証→書く→メタ更新」の4段階で回す。

以上が、LLM開発者視点で「スキルを育てるスキル」のコンテキストループをどう回すか、および何が性能を落とし・引っ張るかの結論である。
