# スキルを育てるスキルとコンテキストループ — 10回弁証法 v2（独立思考版）

**目的**: 前回の出力を一切参照せず、独立した思考でエージェントスキルの性能の本質とコンテキストループの回し方を弁証法で導出する。

---

## ラウンド 1：スキルとは何か

| | |
|---|---|
| **テーゼ** | スキルとは「エージェントに特定ドメインの手順を教える文書」であり、情報量が多いほど有能になる。 |
| **クリティカルな問い** | LLMのattention機構において、コンテキスト内の情報が増えると各トークンへの注意配分はどうなるか？ |
| **アンチテーゼ** | LLMは情報量に比例して賢くなるのではない。コンテキストが長くなるとattentionが拡散し、中間部分の情報が事実上無視される（Lost in the Middle問題）。スキルは「attentionを正しい場所に集中させる装置」であるべき。 |
| **合** | スキルの本質は「知識の量」ではなく「注意の誘導」。LLMの限られたattention予算を、最も重要な手順に集中させるフォーカス装置として設計すべき。 |

## ラウンド 2：コンテキストループの「回す」とは何か

| | |
|---|---|
| **テーゼ** | コンテキストループとは「読む→書く」のイテレーション。回数を増やせばスキルは成熟する。 |
| **クリティカルな問い** | 回数の問題なのか？各ループで「何が変わったか」を検知できなければ、同じ情報を読み書きしているだけではないか？ |
| **アンチテーゼ** | ループの本質は回数ではなく差分の検知と統合。実世界で試した結果（成功/失敗）がフィードバックとしてスキルに流れ込み、スキルの形を変える。差分のないループは空転。 |
| **合** | コンテキストループの1周は「読む→実行して差分を得る→差分をスキルに統合する」。差分のない周回は性能改善に寄与しない。差分がトリガーであり、差分がなければ回す必要すらない。 |

## ラウンド 3：性能を「最も落とす」ものは何か

| | |
|---|---|
| **テーゼ** | 性能を落とすのは「情報の不足」。スキルに書いてないことはエージェントが知りようがない。 |
| **クリティカルな問い** | 本当に不足が最大の敵か？不足ならLLMは一般知識で補完できるが、矛盾した指示や曖昧な指示の場合、補完のしようがないのでは？ |
| **アンチテーゼ** | LLMにとって最も有害なのは矛盾と曖昧さ。ある箇所で「Aせよ」別の箇所で「Bせよ」と書いてあるとき、LLMはどちらに従うかを確率的に選ぶ。出力の不安定性を生む。情報の不足よりも、矛盾・冗長・曖昧さの方が指示追従性を根本的に破壊する。 |
| **合** | 性能を最も落とす本質は「矛盾する指示の共存」。LLMの確率的デコードが「どちらに従うべきか」をランダムに選ぶ結果を生み、実行品質の再現性を破壊する。 |

## ラウンド 4：性能を「最も引っ張る」ものは何か

| | |
|---|---|
| **テーゼ** | ラウンド3の裏返しで、一貫した単一の指示が性能を最も引っ張る。 |
| **クリティカルな問い** | 一貫していても、その指示が「間違って」いたら？実世界で機能しない一貫した指示と、少し揺れるが実世界に適応する指示、どちらが性能が高いか？ |
| **アンチテーゼ** | 一貫性だけでは不十分。スキルは静的な文書だが、世界は動く。実世界とのグラウンディング（検証で正しさを確認し続ける仕組み）がなければ、一貫して間違い続けるだけ。 |
| **合** | 性能を最も引っ張る本質は「一貫性 × グラウンディング」。矛盾のない指示体系を維持しつつ、検証で実世界とのずれを修正し続ける。片方だけでは不十分で掛け算。 |

## ラウンド 5：LLMの「読み方」から見たスキル構造

| | |
|---|---|
| **テーゼ** | スキルの構造は人間が読みやすければよい。見出し・箇条書き・表で整理する。 |
| **クリティカルな問い** | LLMは人間と同じ「読み方」をするのか？LLMにとって本当に効くスキル構造とは何か？ |
| **アンチテーゼ** | LLMはテキストを先頭と末尾を重視し、中間は注意が薄れる。具体的なパターン（入力→出力の例）に対して指示追従性が最も高く、抽象的な原則文には従いにくい。人間の読みやすさとLLMの指示追従性は一致しない。 |
| **合** | スキル構造はLLMの認知特性に合わせるべき。(1) 最も重要な指示を先頭と末尾に置く、(2) 原則より具体例を優先（入力→出力のペア）、(3) 中間部は参照リンクで逃がし本体に詰めない。 |

## ラウンド 6：「育てる」の逆 — スキルの死

| | |
|---|---|
| **テーゼ** | スキルは書いたら存在し続け、少なくとも害にはならない。 |
| **クリティカルな問い** | 古くなったスキルが retrieval に選ばれたとき何が起きるか？古い正しさは現在の間違いになるのでは？ |
| **アンチテーゼ** | 古くなったスキルは嘘をつく。エージェントが古い手順に従い、現在の環境で失敗する。スキルに書いてある以上、エージェントは「これが正しい」と確信して従う。古いスキルは不在より有害。 |
| **合** | スキルには賞味期限がある。検証に失敗したスキルは無効化（アーカイブ）するプロセスが必要。「育てる」には「殺す」（アーカイブ・削除する判断）も含まれる。 |

## ラウンド 7：コンテキストループの「主体」問題

| | |
|---|---|
| **テーゼ** | エージェントがスキル更新のたびに skill-growing を読むと書いてあるから、ループの主体はエージェント。 |
| **クリティカルな問い** | エージェントは「スキルを更新して」と言われないと動かない。誰がその起動トリガーを引くか？ |
| **アンチテーゼ** | 現状のループは人間が「育てて」と言わなければ回らない。ループの起点は人間であり、エージェントは「回す力」を持っていない。検証失敗をエージェントが自動で検知してスキル更新を提案する仕組みがなければ、ループは人間の気まぐれに依存する。 |
| **合** | 真のコンテキストループには「自動トリガー」が必要。理想は (1) 検証スクリプトの定期実行、(2) 失敗時にエージェントへ通知、(3) エージェントが skill-growing の手順で自律修正。現状はまだ (1)(2) が欠けている。 |

## ラウンド 8：スキル間のネットワーク効果

| | |
|---|---|
| **テーゼ** | 各スキルは独立して完結すべき。依存が増えると壊れやすい。 |
| **クリティカルな問い** | 完全に独立したスキル群は、組み合わせたときにシナジーを生むか？ |
| **アンチテーゼ** | スキルの真価はネットワーク効果にある。連鎖したとき初めて複雑なタスクが処理できる。独立性の追求は組み合わせの可能性を殺す。 |
| **合** | スキルは「疎結合だが組み合わせ可能」であるべき。各スキルの入力と出力の型を明示し、スキル間のインターフェースを定義する。内部は独立、入出力の契約で連携。 |

## ラウンド 9：「暗黙知」の扱い

| | |
|---|---|
| **テーゼ** | スキルに書けることだけがスキルの性能に寄与する。暗黙知は形式化するか捨てる。 |
| **クリティカルな問い** | LLMが既に知っていることをスキルに書く意味はあるか？ |
| **アンチテーゼ** | LLMが既に知っていることをスキルに書くのはトークンの無駄遣いであり性能を落とす。スキルに書くべきは「LLMの一般知識にないドメイン固有の暗黙知」だけ。 |
| **合** | スキルの価値密度 = ドメイン固有知識の濃度。育てるとは「LLMが知らないことだけを残す濃縮プロセス」。LLMが進化して一般知識に吸収された内容はスキルから削除してよい。 |

## ラウンド 10：この弁証法自体の限界

| | |
|---|---|
| **テーゼ** | 10回の弁証法で十分な結論が出た。これをスキルに書けばよい。 |
| **クリティカルな問い** | この弁証法自体が一度のセッションの思考であり、実世界での検証を経ていない。グラウンディングされていない結論をスキルに書くことは、ラウンド4で自ら批判した構造と矛盾しないか？ |
| **アンチテーゼ** | 矛盾する。この結論は仮説であり、検証されるまでスキル本体に書くべきではない。 |
| **合** | 弁証法の結論は仮説として references に置き、運用で検証されたものだけを SKILL.md に昇格させる。これ自体がコンテキストループの正しい回し方を体現する。 |

---

## 最終結論

### 性能を最も落とす本質（深い順）

1. **矛盾する指示の共存** — LLMの確率的デコードが「どちらに従うか」をランダムに選ぶ。再現性と信頼性を根本から破壊する。
2. **グラウンディングの欠如** — 検証なしの更新は「一貫して間違い続ける」状態を作る。古いスキルは不在より有害。
3. **attentionの浪費** — LLMが既に知っていること、冗長な説明をスキルに書くこと。本当に重要な指示へのattentionを薄める。

### 性能を最も引っ張る本質（深い順）

1. **一貫性 × グラウンディングの掛け算** — 矛盾のない指示体系を維持しつつ、検証で実世界とのずれを修正し続ける。
2. **ドメイン固有知識の濃縮** — LLMの一般知識にない情報だけを高密度で残す。
3. **差分駆動のループ設計** — 差分がないなら回さない、差分があるなら必ず回す。自動トリガーがこれを可能にする。

### コンテキストループの回し方

| 要素 | 現状 | 理想 |
|------|------|------|
| 起点 | 人間が「育てて」と言う | 検証失敗の自動検知 |
| 1周の定義 | 読む→検証→書く→メタ更新 | 差分検知→読む→検証→統合→メタ更新→再検証 |
| 書く内容 | 知見の追記 | 矛盾の排除 + 既知の削除 + 知見の追記 |
| 終了条件 | 書き終わったら | 再検証に通ったら |
| スキルの死 | 考慮なし | 検証不合格が続くスキルをアーカイブ |
