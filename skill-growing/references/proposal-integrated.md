## 1. 4モデルの「深層構造」— なぜその結論に至ったか

### 1.1 認知パターンの比較

| モデル | 思考の出発層 | 本質的に見ていたもの | LLMとしての自己認識 |
|--------|-------------|---------------------|-------------------|
| v1（Auto） | 手順層 | 「何をするか」のリスト | 浅い。自分がどう読むかに言及せず |
| v2（Opus） | 概念層 | attentionと差分。矛盾が確率的デコードで破壊する | 深い。自分の確率的選択の弱点を自覚 |
| v3（GPT） | 工学層 | パイプラインの歩留まり。adherenceのサイレント劣化 | 最も実践的。「従われない指示は存在しないのと同じ」 |
| v4（Gemini） | 存在論層 | 引力場と意味密度。チャネル間干渉 | 最も抽象的だが「二重の読者」は唯一の実用的洞察 |

### 1.2 4モデルが共通して告白していること

4モデルとも、「スキル文書を自分がどう処理するか」を正直に語っている。通底するメッセージ：

> **「書いてあることの多くに実は従えない」**

- v2：矛盾があるとランダムに選ぶ
- v3：制約が増えるほど各制約への追従確率が下がる
- v4：量を増やすと各文の影響力は弱まる

この告白から導かれる原則：

> **スキルを育てるとは「足す」ことではなく「研ぐ」こと。研ぐとは、LLMが確実に従える少数の指示だけを残し、残りを参照に退避させること。**

### 1.3 モデル間で異なる「最大の敵」

| モデル | 最大の敵 | 実務的な意味 |
|--------|---------|-------------|
| v2 | 矛盾する指示の共存 | 「gws で書く」と「API のみで書く」の共存 → 出力が毎回ブレる |
| v3 | Adherence のサイレント劣化 | 制約10個のスキル → 各制約に従う確率が下がり「なんか微妙な結果」が増える |
| v4 | チャネル間干渉 | Sheets読み・Sheets書き・Docs操作が1スキルに混在 → 推論方向がゆがむ |

**3つとも実在する脅威**であり、安定化の際は3つすべてをチェックする。

## 2. 不安定スキル（gws/Sheets/Docs）の診断

### 2.1 v3 の 4段階パイプラインによる診断

| 段階 | 現状の問題 | 根拠 |
|------|-----------|------|
| **Selection** | `gws-params-encoding` と `sheet-api-update` が別スキル。「スプレッドシートに書き込みたい」とき、どちらが選ばれるか不安定 | v4「チャネル間干渉」 |
| **Injection** | `sheet-api-update` は88行。認証の優先順・4種のスクリプト・エラー対処が全部入っている | v3「injection 診断：他の文脈を圧迫していないか」 |
| **Adherence** | 「書き込みは API のみ」「gws でも読取可」— 同一ドメインで2つの方法が共存。毎回「今回はどちらで？」と推論が迷う | v2「矛盾する指示の共存」 |
| **Execution** | Windows 特有のエスケープ問題。Docs 対応の欠如。検証スクリプトが params-encoding にしかない | v3「ドリフト検知装置の不在」 |

### 2.2 最大のボトルネック

**Adherence のサイレントな劣化**（v3）。

「読みは gws も OK、書きは API だけ、Docs はまだ NG」という条件付き分岐が、エージェントに毎回推論させ、**推論方向がブレる**（v4 のチャネル間干渉）。

## 3. 提案

### 3.1 提案A：即座に実行すべきこと —「研ぐ」フェーズ

#### A-1. gws 関連スキルを「推論方向」で再編成する

**現在**: `gws-params-encoding`（エンコーディング問題）と `sheet-api-update`（API 書き込み）が別スキル。

**提案**: **1スキル = 1つの推論方向** に整理。「Google Workspace への読み書き」を統合した判定表を置き、内部で3行の分岐にする：

```
① スプレッドシート書き込み → scripts/requirements-to-sheet.mjs（API）
② スプレッドシート読取 → gws values get
③ Google Docs → 未対応。エラーを出して止まれ。
```

params-encoding の詳細は references に退避。

**根拠**: v4「スキルの境界 = 推論の方向が変わる地点」。読み書きの方法選択は同じ推論方向であり、別スキルに分けるとチャネル間干渉が起きる。

#### A-2. Adherence を上げる「3つの制約」に研ぐ

SKILL 本体にはこの3行の判定表**だけ**を残す。認証の詳細・エスケープの仕組み・エラー対処はすべて references に退避。

**根拠**: v3「制約は3つに絞れ。書かないことが adherence を上げる」。

#### A-3. 検証スクリプトを「書き込み」にも追加する

- 現状: `gws-params-encoding` にはテストがあるが、`sheet-api-update` のスクリプト群には検証がない
- 提案: `scripts/sheet-api-smoke-test.mjs` を作成。「テストシートに1行書き込み → 読み返し → 一致確認」を自動化。スキルの検証手順に追加
- **根拠**: v1〜v4 全版共通の検証必須化

### 3.2 提案B：skill-growing 自体の進化 —「差分駆動 + 置換型ループ」

#### B-1. ループの手順を更新する

**現在の手順**:
> 読む → 検証する → 書く（追記）→ メタを更新する

**提案する新しい手順**:
> 1. **読む** — skill-growing + 該当スキル + references
> 2. **検証する** — 検証スクリプト実行。失敗なら修正してから先へ
> 3. **差分を判定する** — 何が変わったか？ 変わってないなら終了（空転防止）
> 4. **置換する** — 追記ではなく、古い記述を新しい記述に**置き換え**。総量は増やさない
> 5. **メタを更新する** — description/トリガーが実態と合っているか確認
> 6. **再検証する** — 置換後に検証が通ることを確認

**根拠**:
- v2「差分のないループは空転」→ ステップ3を追加
- v3「追記型ループは振動する」→ ステップ4を「置換」に変更
- v3/v4「再検証でグラウンディング」→ ステップ6を追加

#### B-2. トリガーの拡張

**現在**: 「スキルを更新して」「スキルを育てて」「検証を追加して」で発動。

**提案**: 以下もトリガーに含める：
- 「〇〇でエラーが出る」「〇〇が動かない」「反映で失敗する」— 問題報告
- 検証スクリプトの失敗 — 差分検知

**根拠**: v2「理想は検証失敗の自動検知」。完全な自動化は基盤が必要だが、トリガーに「失敗・問題報告」を含めることで差分駆動に近づく。

#### B-3. 4軸簡易診断をループに組み込む

検証の前後に、v3 の 4軸で「なぜ性能が落ちているか」を切り分ける：

| ステップ | 問い | 操作 |
|----------|------|------|
| Selection 診断 | 正しい場面で選ばれているか？ | description/トリガー調整 |
| Injection 診断 | 他の文脈を圧迫していないか？ | 冗長部分の圧縮・参照分離 |
| Adherence 診断 | 書いてあることに従っているか？ 制約は3つ以内か？ | 弱い制約の削除、表現強化 |
| Execution 診断 | 手順は現在の環境で動くか？ | 検証実行、手順の置換 |

### 3.3 提案C：中長期 —「賞味期限」と「観測の蓄積」

#### C-1. 賞味期限の導入

各スキルのフロントマターに `last_verified: YYYY-MM-DD` を追加。30日以上検証されていないスキルは、エージェントが使用時に「このスキルは30日以上検証されていません。検証を先に実行しますか？」と警告する一文を skill-growing に追加。

**根拠**: v2/v4「古いスキルは不在より有害」「スキルには賞味期限がある」

#### C-2. 失敗記録の構造化

スキル使用時にエラーが出たら、`references/failure-log.md` に日付・エラー内容・対処を記録するルールを追加。これが v2 の「差分」のトリガーになる。

#### C-3. 証拠駆動の昇格

references に「仮説プール」を維持。運用で複数回検証に成功した原則だけを SKILL.md に昇格。逆に、適用しても失敗が続く原則は仮説プールに戻すか注釈する。

**根拠**: v2「結論をそのまま SKILL に載せない」+ v3「1つの原則だけ昇格」+ v4「外部観測のみで判定」

## 4. 判断の根拠一覧

| 判断 | 根拠となるモデルの洞察 | なぜそれを採用したか |
|------|----------------------|---------------------|
| 「研ぐ（引き算）」を第一に | v3「書く量が増えるほど追従確率が下がる」、v4「何を書かないかで決まる」 | 4モデル中最も一致度が高い結論。attention 機構から演繹的に導かれ、反証が困難 |
| パイプライン4段階で診断 | v3 の selection × injection × adherence × execution | 唯一「測定可能な軸」を提示。問題がどこにあるかを特定できるフレームワーク |
| 推論方向でスキル境界を引く | v4「スキルの境界 = 推論の方向が変わる地点」 | gws が不安定な原因（読みと書きで推論方向が分裂）を説明できる唯一の概念 |
| 追記 → 置換 | v3「追記型ループは振動する」 | v1は追記を前提、v3・v4が明示的に否定。gws が「足しすぎて不安定」という現象と整合 |
| 差分駆動 | v2「差分のないループは空転」 | 何も変わっていないのにスキルを触るとノイズが増えるだけ |
| 弁証法の結論は references に留める | v2〜v4 全版 | 未検証の仮説を SKILL に直接載せると v3 の自己矛盾（「3つに絞れと言って10個書く」）に陥る |
