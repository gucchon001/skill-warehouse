# 開発計画の更新（ドキュメント＋整合確認）

**エージェントへの指示**: **第1層の `description` だけで短絡しない**。

## いつ使うか / When to use

- **主トリガー**: 「**開発計画を更新して**」（同義: 計画のブラッシュアップ・最新化）
- マイルストーン／スコープ／技術スタック変更後の計画の現状化
- 仕様の鮮度、**コードと doc の矛盾**の洗い出し

## 手順 / How to use

### 1. 開発計画の正本を特定する

`docs/**/06-*-project-plan.md`, `**/PLAN.md`, `**/ROADMAP.md`, `docs/10-product/*plan*`, `migration-pack/**/README.md` の計画節などを **検索**（複数候補は `docs/README` やユーザー確認）。専用ファイルが無ければ `README` の Roadmap 節を正とするか **`docs/development-plan.md` 新設**を提案。

### 2. 関連ドキュメントを読む

仕様索引（例: `docs/README.md`）。プロジェクトに **spec-creation** や要件／機能の正本があれば Read。突合せの観点は **`references/parity-checklist.md`**。

### 3. コード側の事実（スコープに合わせる）

エントリ・主要ルート、環境変数名、依存の版、削除済み機能の残骸有無。詳細観点は **`parity-checklist.md`**。

### 4. 計画ドキュメントを更新する

日付・フェーズ・マイルストーン、スコープ外／完了／延期、技術スタックと実装の一致、仕様側の決定への **「参照: …」** リンク。

### 5. 報告（必須）

1. 更新したファイルと変更要約（章単位可）  
2. 整合結果: 問題なし／軽微／要対応（該当はファイル・見出しで指す）  
3. 提案: 他 doc やコードのどちらを直すか  
4. **未確認** 範囲を明示  

## 他スキル

- **仕様の章立て・ルート表の網羅パリティ**はプロジェクトの **spec-creation**（または同等）を優先。本スキルは **計画レベル**の矛盾とマイルストーン整合が主。
- 長セッション直後に文脈を残すだけ、または「**新しいチャットする準備**」なら **`thread-handoff`**（HANDOFF 正本。旧名: cursor-thread-handoff）。

## Troubleshooting

### エラー: どれが開発計画か分からない

**原因**: ファイル名が分散、README だけに記載。

**対処**: `docs` を `plan`, `roadmap`, `migration`, `WBS` で検索し候補を列挙して 1 つに絞る、または `docs/development-plan.md` を新設。

### エラー: 仕様は新しいがコードが古い

**原因**: 仕様先行で実装が未追従。

**対処**: 報告で正本を明示し、ギャップを Issue 文案やチェックリストに落とす。コード変更は別指示があれば実施。

### エラー: 一度に監査しすぎる

**原因**: リポジトリ全体スコープになった。

**対処**: **計画が触れているマイルストーン・機能名**に限定し、残りは未確認リストへ。
